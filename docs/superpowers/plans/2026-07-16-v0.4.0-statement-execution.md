# v0.4.0 Statement Execution Implementation Plan

**Goal:** Add `gs` in Query Normal mode to execute and highlight the statement under the cursor while preserving whole-buffer `Enter` and `Ctrl+Enter` behavior.

**Architecture:** Keep one lossless, backend-aware lexical analysis seam in `src/sql.rs`. It classifies protected bytes, extracts executable words, chooses statement ranges, rejects compound definitions that cannot be split safely, and determines both whether the depth-zero main statement returns rows and whether it is safe to wrap for pagination. `App` and the PostgreSQL/MySQL adapters consume that shared classification. No parser dependency or block parser is added.

**Release:** `0.4.0`, with no config or history format migration.

## Behavioral contract

### Statement selection

- `statement_at_cursor(&str, (usize, usize), DbType)` returns an exact UTF-8 byte range plus one-based ordinal and total.
- Cursor columns are character indexes, then mapped to UTF-8 byte offsets.
- If any unprotected semicolon exists, semicolons are the only boundaries. Otherwise, blank lines outside protected regions split statements.
- Single and double quotes, MySQL backticks, SQLite brackets, comments, and PostgreSQL dollar blocks protect internal semicolons and blank lines.
- PostgreSQL block comments nest. MySQL `#` starts a comment only on MySQL; MySQL `--` starts a comment only when followed by whitespace or a control byte.
- MySQL executable comments (`/*! ... */`) protect their internal semicolons while their executable words still participate in safety and query classification.
- Doubled quotes are honored. Backslashes escape MySQL strings and PostgreSQL `E'...'`, but remain literal for SQLite quotes and MySQL backticks.
- PostgreSQL dollar openers require an identifier boundary and accept valid non-ASCII tag bytes.
- Unterminated protected regions fail closed. Empty and ordinary-comment-only ranges produce no statement.

### Compound-definition safety

The scanner deliberately does not parse procedural blocks or client delimiters. It rejects recognized definition-leading forms before their inner semicolons can become executable ranges:

- SQLite `[EXPLAIN [QUERY PLAN]] CREATE [TEMP|TEMPORARY] TRIGGER`;
- MySQL `CREATE` procedure/function/trigger/event, including object words inside executable comments;
- MySQL compound `ALTER EVENT ... DO [label:] BEGIN ... END`; and
- PostgreSQL `CREATE [OR REPLACE] FUNCTION|PROCEDURE ... BEGIN ATOMIC ... END`.

Metadata statements such as MySQL `SHOW CREATE TRIGGER|PROCEDURE|FUNCTION|EVENT` are not definitions and remain executable. A compound definition anywhere in the buffer, including after a safe statement, fails the scan.

### Query dispatch

- `gs` queues only the selected source slice through the existing async query path.
- Normal `Enter`, Insert `Ctrl+Enter`, and Explorer `s` remain fresh whole-query execution paths.
- Fresh execution resets pagination. Only `PgUp`/`PgDn` page replay preserves the chosen offset.
- Query history records exactly the SQL sent to the adapter.
- The shared row classifier recognizes top-level `SELECT`, `VALUES`, and `TABLE`.
- For `WITH`, it skips CTE bodies plus PostgreSQL `SEARCH`/`CYCLE` clauses and classifies the depth-zero main statement: row-producing `SELECT`/`VALUES`/`TABLE` paginate; `INSERT`/`UPDATE`/`DELETE` do not.
- A PostgreSQL row-producing query with an `INSERT`/`UPDATE`/`DELETE` CTE still returns rows, but is not wrapped for pagination because data-modifying CTEs must remain attached to the top-level statement.
- Scanner feedback lives in `status_message`, not `QueryStatus`. If an older database error exists, the status bar composes it with newer scanner feedback so the feedback stays visible.

### Highlight lifecycle

- The selected range uses `theme.selection_bg` without replacing token foreground colors or modifiers.
- The highlight survives query success and database failure.
- Cursor movement, edit, paste, or another execution invalidates it.

## Files

| Path | Responsibility |
|---|---|
| `src/sql.rs` | Shared backend-aware byte classification, word analysis, statement selection, compound safety, row-returning classification, and pagination-wrap safety. |
| `src/app.rs` | Selected-statement state, fresh/page queueing, shared pagination classification, status composition, and range-aware rendering. |
| `src/db/postgres.rs`, `src/db/mysql.rs` | Consume the shared query classification; MySQL also derives transaction keywords from its words. |
| `src/mode/editor/normal.rs` | `gs` prefix dispatch and binding. |
| `src/mode/editor/insert.rs`, `src/mode/explorer.rs`, `src/mode/results.rs` | Fresh whole-query and page-replay routing. |
| `tests/statement_scanner.rs` | Dialect boundaries, malformed input, compound safety, cursor mapping, and blank-line fallback. |
| `tests/query_execution.rs`, `tests/statement_highlight.rs` | Dispatch, status, selection, and rendering lifecycle. |
| `tests/history_app_hook.rs`, `tests/results_pagination.rs` | Exact history slice and pagination behavior. |
| `README.md`, `CONTEXT.md`, `CHANGELOG.md` | User, domain, and release documentation. |

## Implementation tasks

1. Add scanner regressions for semicolon precedence, all protected regions, dialect comment rules, UTF-8 cursor columns, malformed input, and compound definitions.
2. Implement the byte classifier and word analysis together in `src/sql.rs`; keep tokenizer highlighting independent.
3. Add query-classification regressions for leading/nested comments, executable comments, row-producing CTEs, and CTE-led DML.
4. Route `gs` through fresh query queueing and all consumers through the shared row classifier.
5. Add selection-aware rendering and invalidation tests.
6. Update user/domain/release documentation and run all release gates.
