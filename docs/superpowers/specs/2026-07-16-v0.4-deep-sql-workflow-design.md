# v0.4 deep SQL workflow train — design spec

**Status:** design approved; document review pending
**Date:** 2026-07-16
**Owner:** tuandzung
**Tags covered:** v0.4.0, v0.4.1, v0.4.2

## 1. Overview

v0.4 deepens sqrit's core TUI workflow without expanding its backend list or adding a CLI mode. Each release ships one independent feature.

| Tag | Feature | Promise |
|---|---|---|
| v0.4.0 | Statement execution | `gs` runs the statement under the cursor without changing existing execution keys. |
| v0.4.1 | Definition viewer | `d` in Explorer loads copy-ready object DDL into a read-only modal. |
| v0.4.2 | Context-aware autocomplete | Qualified completion resolves aliases and namespaces and inserts correctly quoted identifiers. |

### Cross-cutting rules

1. Each tag must remain independently shippable and revertible.
2. SQLite, PostgreSQL, and MySQL remain first-class.
3. V1–V11 continue to hold. Database work crosses the `Database` trait and runs off the UI thread.
4. Existing config and history files remain compatible.
5. v0.4.0 and v0.4.1 add no dependency. v0.4.2 may add the pure-Rust `sqlparser` crate.
6. New mode bindings come from `ModeHandler::bindings()`.

### Out of scope

- Visual mode and selected-text execution
- Destructive-query confirmation
- CLI query mode
- CTE, derived-table, or nested-scope completion
- Clause-aware completion and completion ranking
- Table and sequence definitions
- MySQL client commands such as `DELIMITER`
- SQL Server `GO`

## 2. Architecture decision

Use a hybrid SQL architecture.

- v0.4.0 uses a small lossless scanner. It must preserve exact editor ranges, tolerate incomplete SQL, and implement sqrit's blank-line fallback.
- v0.4.1 uses native metadata queries behind the existing database seam.
- v0.4.2 uses `sqlparser` only to resolve top-level relations and aliases. The parser never owns editor ranges or syntax highlighting.

### Alternatives rejected

- **Extend sqrit's tokenizer for every feature.** This avoids a dependency but turns alias resolution into a growing hand-written parser.
- **Use `sqlparser` everywhere.** A strict AST parser handles incomplete editor buffers poorly. Its source spans also remain incomplete, so it cannot reliably drive exact statement highlighting.

## 3. v0.4.0 — statement execution

### Goal

Run one statement from a multi-statement buffer without changing whole-buffer execution.

### User flow

- `gs` in Query Normal runs the statement containing the cursor.
- `Enter` in Query Normal still runs the whole buffer.
- `Ctrl+Enter` in Query Insert still runs the whole buffer.
- The hint bar and help overlay list `gs` from the same Normal-mode binding table.

### Statement model

Add a source-range result to `src/sql.rs`:

```rust
pub struct StatementRange {
    pub range: std::ops::Range<usize>,
    pub ordinal: usize,
    pub total: usize,
}

pub fn statement_at_cursor(
    sql: &str,
    cursor: (usize, usize),
    backend: DbType,
) -> Result<Option<StatementRange>, StatementScanError>;
```

`range` contains UTF-8-boundary byte offsets into the original buffer. The caller executes the original slice; the scanner never reformats SQL.

### Boundary rules

1. Scan single-quoted strings, double-quoted identifiers/strings, backtick identifiers, SQLite bracketed identifiers, line comments, block comments, and PostgreSQL dollar-quoted bodies.
2. Honor doubled quote escapes. Backslash escaping is backend- and quote-aware: SQLite treats it literally, MySQL default string quoting honors it, PostgreSQL honors it for `E'...'`, and MySQL backtick identifiers treat it literally.
3. Handle PostgreSQL nested block comments and backend-specific MySQL comments: `#` is MySQL-only, and `--` requires following whitespace or a control byte on MySQL. MySQL executable comments (`/*! ... */`) protect internal semicolons while exposing their executable words to safety and query classification.
4. Before producing ranges, conservatively reject definition-leading SQLite triggers, MySQL procedure/function/trigger/event definitions (including compound `ALTER EVENT`), and PostgreSQL SQL-standard routines with `BEGIN ATOMIC` bodies. Their compound bodies contain executable inner semicolons; v0.4.0 fails closed instead of parsing blocks or supporting `DELIMITER`. Metadata statements such as MySQL `SHOW CREATE TRIGGER` remain executable.
5. Otherwise, if the buffer contains any semicolon outside protected regions, use semicolons as the only boundaries.
6. Otherwise, split on blank lines outside protected regions.
7. Remove leading and trailing blank space from executable ranges but retain a terminating semicolon.
8. Ignore empty and comment-only ranges.
9. A cursor inside a statement selects that statement.
10. A cursor on a separator or trailing space selects the nearest preceding executable statement.
11. Space before the first statement selects the first statement.
12. An unterminated string, block comment, quoted identifier (including SQLite brackets), or dollar block returns an error. `gs` executes nothing.

The semicolon-first rule intentionally matches sqlit's behavior for statements the scanner can safely separate. For example, a single real semicolon switches the entire buffer out of blank-line mode. The compound-DDL rejection above is the deliberate exception.

### Execution and state

`NormalHandler` extends its existing `g` prefix with `s`. On success it:

1. stores the selected range and ordinal on `App`;
2. sets `pending_query` to the selected source slice;
3. sets `status_message` to `running statement N/M`; and
4. enters the existing async query path.

The existing path sets `last_query`, records history, paginates row-returning queries, handles cancellation, and rejects stale query results. History therefore stores only the SQL sent to the adapter.

Every fresh execution (`gs`, Normal `Enter`, Insert `Ctrl+Enter`, and Explorer `s`) resets pagination. `PgUp` and `PgDn` are explicit page replays and preserve their selected offset. App and PostgreSQL/MySQL adapters share the scanner's backend-aware row classifier. It recognizes top-level `SELECT`, `VALUES`, and `TABLE`; for `WITH`, it skips CTE bodies by parenthesis depth and classifies the depth-zero main statement, so CTE-led `INSERT`, `UPDATE`, and `DELETE` do not use the paginated row path.

Scanner feedback belongs to `status_message`, not `QueryStatus`. A scanner error or empty selection must not overwrite an already-running database query. When an older `QueryStatus::Error` exists, the status bar composes it with newer scanner feedback instead of hiding that feedback. `running statement N/M` remains while the selected query is queued/running, then clears on its matching completion, on edit/paste invalidation, or when any non-`gs` execution is queued.

### Highlight

While rendering the editor, split syntax spans where they intersect the selected range. Apply `theme.selection_bg` to the selected characters and preserve token foreground colors and modifiers.

The highlight survives success and error. Cursor movement, an editor mutation, or another execution clears it. Exact character coverage prevents one statement from highlighting a neighbor on the same line.

### Errors

- Empty or comment-only buffer: `no statement at cursor`.
- Unterminated protected region: identify the construct in the status bar without changing database query state.
- Recognized compound SQLite/MySQL/PostgreSQL definition: report that it cannot be scanned safely and execute nothing.
- Database error: use the existing query error path and retain the selected range.

### Tests

- Semicolon splitting and blank-line fallback
- Global semicolon precedence over blank lines
- Semicolons and blank lines inside every protected region
- PostgreSQL anonymous and tagged dollar blocks
- Backend-specific comment rules
- SQLite bracket protection, backend-aware backslashes, and doubled quotes
- Conservative fail-closed SQLite triggers, MySQL compound definitions, and PostgreSQL `BEGIN ATOMIC` routines; MySQL `SHOW CREATE` metadata remains allowed
- Separator, leading-space, and trailing-space cursor behavior
- Empty/comment-only ranges
- Fail-closed malformed constructs
- `gg` and `gs` prefix dispatch
- Existing whole-buffer key behavior
- Fresh-query pagination reset, page-replay offset preservation, and shared CTE-aware row-returning detection
- Scanner feedback and `running statement N/M` lifecycle while another query is active
- Exact background range and preserved syntax style
- Highlight clearing and persistence
- History contains only selected SQL

## 4. v0.4.1 — definition viewer

### Goal

Inspect and copy object definitions without replacing the query buffer or results.

### Supported objects

| Kind | SQLite | PostgreSQL | MySQL |
|---|---:|---:|---:|
| View | yes | yes | yes |
| Materialized view | — | yes | — |
| Index | yes | yes | yes |
| Trigger | yes | yes | yes |
| Function | — | yes | yes |
| Procedure | — | yes | yes |

Tables and sequences remain out of scope because PostgreSQL has no equivalent single native table-definition function and sequence recreation needs more metadata than v0.3 stores.

### Object identity

Introduce an adapter-neutral reference:

```rust
pub struct ObjectRef {
    pub namespace: String,
    pub kind: ObjectKind,
    pub name: String,
    pub relation: Option<String>,
    pub identity_arguments: Option<String>,
}
```

`relation` identifies the owning table for indexes and triggers; it is `None` for other kinds. `identity_arguments` distinguishes overloaded PostgreSQL routines and is `None` elsewhere. `TreeItem::Object` carries this reference instead of discarding metadata while flattening the schema tree.

PostgreSQL routine introspection also selects `pg_get_function_identity_arguments(oid)`. Explorer renders overloaded routines as `name(argument_types)` while retaining the bare name for lookup and quoting.

### Database seam

Extend `Database`:

```rust
async fn object_definition(
    &self,
    object: &ObjectRef,
) -> anyhow::Result<Option<String>>;
```

Adapters return `None` when the backend cannot produce a definition for that object. They return an error for permission failures, stale objects, and malformed metadata.

### Adapter behavior

- **SQLite:** read the normalized `CREATE` statement from `sqlite_schema.sql`.
- **PostgreSQL:** use `pg_get_viewdef`, `pg_get_indexdef`, `pg_get_triggerdef`, and `pg_get_functiondef`. Wrap view and materialized-view query bodies in qualified `CREATE` statements.
- **MySQL:** use `SHOW CREATE VIEW`, `SHOW CREATE TRIGGER`, `SHOW CREATE FUNCTION`, and `SHOW CREATE PROCEDURE`. For an index, use its owning `relation` to read `SHOW CREATE TABLE`, select the server-rendered index clause, and emit `ALTER TABLE <qualified table> ADD <clause>;`. MySQL has no `SHOW CREATE INDEX` statement.

All generated wrappers use the existing backend quote helpers. Output ends with a semicolon and preserves native clauses.

### Async flow

Add `DefinitionViewerState` with `Loading`, `Ready(String)`, and `Error(String)` content states, plus scroll offset and the selected `ObjectRef`.

1. `d` on a supported Explorer object opens `Mode::DefinitionViewer` immediately.
2. `App` increments a definition request ID and spawns `object_definition()`.
3. `AsyncResult::DefinitionLoaded` carries the ID and result.
4. `drain_async_results()` ignores stale IDs after close or reopen.
5. The modal replaces `Loading…` with DDL or an error.

Definitions are fetched on every open. v0.4.1 adds no cache.

### Modal

Use the Cell Viewer rectangle, border, theme, and scrolling conventions without merging the two modes.

- `j/k`: scroll
- `y`: copy ready DDL through `ClipboardWriter`
- `Esc`: restore Explorer

`ModeHandler::bindings()` supplies modal hints and help text. `y` is inert until content is ready.

### Tests

- `ObjectRef` extraction from every supported Explorer node
- Owner-relation identity for indexes and triggers
- PostgreSQL overload labels and exact routine lookup
- Adapter definition retrieval for every supported kind
- Qualified and escaped identifiers
- MySQL index-clause extraction
- Copy-ready wrappers and terminators
- Loading, ready, and error rendering
- Stale async result rejection
- Scroll, copy, close, and retry-by-reopen behavior
- Unsupported group, column, table, and sequence nodes

## 5. v0.4.2 — context-aware autocomplete

### Goal

Resolve qualified relations and columns without turning autocomplete into a full SQL language server.

### Dependency

Add `sqlparser = "0.62"` (Apache-2.0, pure Rust). Select `PostgreSqlDialect`, `MySqlDialect`, or `SQLiteDialect` from the active connection.

### Candidate model

Replace string-only candidates with:

```rust
pub struct CompletionCandidate {
    pub label: String,
    pub detail: Option<String>,
    pub insert_text: String,
}
```

The popup renders `detail` as secondary text. Existing keyword, type, and bare-prefix completion uses the same structure.

### Resolution flow

1. Reuse v0.4.0's scanner to isolate the current statement.
2. Read the identifier path immediately before the cursor.
3. For alias resolution, replace the incomplete target after the final dot with a harmless placeholder and parse the statement.
4. Walk only top-level `FROM` and `JOIN` relations.
5. Map explicit and implicit aliases to namespace-rooted schema objects.
6. Resolve completion against `SchemaInfo`.

Supported cases:

- `u.` after `FROM users u` or `FROM public.users AS u` returns columns from `users`.
- `o.` after `JOIN orders o` returns columns from `orders`.
- `public.` returns relations in `public`.
- `public.users.` returns columns from that relation.
- An alias shadows a namespace with the same name.
- An unknown qualifier returns no candidates.

CTEs, derived tables, correlated subqueries, nested alias scopes, clause-aware filtering, routines, fuzzy ranking, and MRU ranking remain out of scope.

### Ambiguous names

Bare relations with the same name remain separate popup rows. `label` holds the relation name and `detail` holds its namespace. Selecting an ambiguous row inserts the qualified path. A unique relation keeps the existing concise insertion.

### Identifier quoting

Add required-only quoting beside the existing always-quote helpers.

- Leave a safe, non-keyword identifier bare.
- Quote reserved words, mixed-case PostgreSQL identifiers, spaces, and special characters.
- PostgreSQL and SQLite use doubled double quotes.
- MySQL uses doubled backticks.
- When the qualifier already exists in the buffer, insert only the final segment.

### Failure behavior

AST parsing is advisory. If the incomplete statement still fails after placeholder insertion, qualified completion returns no candidates. Editing, highlighting, and existing bare completion continue normally.

### Tests

- Explicit and implicit aliases
- Top-level joins
- Namespace and three-part qualification
- Alias-over-namespace precedence
- Duplicate relation labels, details, and insertion text
- Required-only quoting for every backend
- Unknown qualifiers
- Parser failure on incomplete or vendor-specific SQL
- CTE and nested scopes remain unresolved
- Popup detail rendering and Tab replacement
- Existing 300 ms trigger and bare completion behavior

The release passes when all three backends support the promised behavior, existing whole-buffer execution remains unchanged, and no feature requires a config migration.

## 7. Reference comparison

sqlit informed four choices:

- explicit current-statement execution rather than changing the default key;
- semicolon-first splitting with blank-line fallback;
- PostgreSQL dollar-block handling; and
- schema-qualified, required-only autocomplete insertion.

sqrit deliberately stops before sqlit's visual modes, full statement-specific completion engine, custom keymaps, and broader connection surface.
