**Status**: approved, ready for implementation planning
**Date**: 2026-05-25
**Owner**: tuandzung
**Tags covered**: v0.3.0, v0.3.1, v0.3.2

## 1. Overview

Three independently-shippable features, one cohesive "UX polish" theme. All additive: no V1–V9 invariant changes, no breaking config or CLI changes (sqrit has no CLI yet — that's v0.4+).

| Tag | Feature | New ADR | Primary modules touched |
|---|---|---|---|
| v0.3.0 | Fuzzy filter (replaces v0.2 substring) | none | `src/filter.rs` (new), `src/results.rs`, `src/mode/results*.rs` |
| v0.3.1 | Hint bar | ADR 7 | `src/app.rs`, `src/mode.rs`, `src/hint_bar.rs` (new), `src/config/mod.rs`, theme TOML |
| v0.3.2 | Schema browser (namespace-aware) | ADR 8 | `src/db/types.rs`, all three adapters in `src/db/`, `src/explorer.rs` |

### Cross-cutting principles
1. Each minor is independently revertable — no shared TOML/code field landed before its release.
2. `~/.sqrit/config.toml` is forward-additive: unknown fields ignored, missing fields default. v0.2.0 binaries reading a v0.3.x config keep working.
3. Help overlay and hint bar share a single source — `ModeHandler::bindings()`. ADR 3 trait-dispatch contract holds.
4. No new system dependencies. New crates only: `nucleo-matcher` 0.3 (MPL-2.0, pure Rust).

### Out of scope (deferred to v0.4+)
- OS keyring credentials; `--password-command`
- URL-based connections; SSH tunnels
- CLI query mode (`-q` / `-f` / `--output`)
- New backends (DuckDB / SQL Server / Snowflake / BigQuery)
- Docker container auto-discovery
- Mock mode
- Definition viewer (`d` on indexes/triggers/functions)
- Lazy per-kind schema introspection
- Editable cell viewer / DML generation
- Marketplace for community themes

## 2. v0.3.0 — Fuzzy filter

### Goal
Replace v0.2 case-insensitive substring filter in Results with subsequence-scoring fuzzy match. Highlight matched characters inline. Keep V6 (current-page-only scope, never the full result set in memory).

### Architecture
Single `nucleo_matcher::Matcher` reused across keystrokes. Filter state lives on `App` (not matcher-owned) so it can re-run on each PgDn-fetched page. `/` flow unchanged from v0.2; semantics behind it change.

### Components

```rust
// src/filter.rs (new)
pub struct FuzzyFilter {
    matcher: nucleo_matcher::Matcher,
    haystack_buf: Vec<char>,
    indices: Vec<u32>,
}

pub struct FilterHit {
    pub row_index: usize,
    pub score: u32,
    pub matches: Vec<(usize, std::ops::Range<usize>)>, // (column index, grapheme range)
}

impl FuzzyFilter {
    pub fn rank(&mut self, result: &QueryResult, query: &str) -> Vec<FilterHit>;
}
```

`ResultsState` (extends existing struct in `src/results.rs`):
- `filter: Option<String>` — live edit buffer and persisted query; `None` means no filter
- `filter_hits: Vec<FilterHit>` — rebuilt on every keypress while in `ResultsFilter` mode

### Data flow

1. `/` → `Mode::ResultsFilter`, initialise an empty `filter`.
2. Each char → update `filter`, call `FuzzyFilter::rank()` over the current page, store hits ordered by score descending.
3. Render: iterate hits in score order. For each hit, walk row columns and consult `FilterHit.matches` to wrap matched grapheme ranges with `Style::default().add_modifier(Modifier::BOLD | Modifier::UNDERLINED).fg(theme.border_focused)`.
4. Enter → keep `filter` and return to `Mode::Results`. Title displays `Results (filter: <query>)`.
5. PgDn → fetch page, re-call `rank()` with the committed query before render.
6. Esc → clear filter state, restore unfiltered view in original row order.

### Library wiring
- `nucleo-matcher = "0.3"` (MPL-2.0).
- `Matcher::new(Config::DEFAULT)` with `Pattern::parse(query, CaseMatching::Ignore, Normalization::Smart)` for case-insensitive, smart-normalised matching.
- Each column is scored independently with `Pattern::indices`; matching ranges retain their column index and row score is the saturating sum of all matching column scores.
- Ranking: descending combined row score; ties broken by original row index for stability.

### Error handling
- `nucleo-matcher` is infallible — no fallible path.
- Empty `filter` → all rows pass through, no highlighting, original order preserved.
- Control characters in pattern stripped before `Pattern::parse`.

### Testing
- Unit `tests/fuzzy_filter.rs`:
  - `usr@bc` matches `user_at_bc.io` with non-zero score
  - Exact substring `bcorp` scores higher than scattered subsequence `bp`
  - Empty query returns full row set in original order, `FilterHit.matches` empty
  - Unicode (CJK) doesn't panic; match ranges align with rendered graphemes
- Integration: existing `tests/results_*.rs` extended — filter persists across simulated PgDn, hits recomputed for the new page.
- Snapshot test for highlight rendering via hand-rolled `Vec<Span>` assertion (avoid pulling `insta` for one test).

### Docs / changelog
- `CONTEXT.md` "Filter (v0.2)" → "Filter (v0.3)": drop current-state "substring" wording; mention `nucleo-matcher`, highlighting, V6 still holds; preserve v0.2 substring behavior as explicitly historical.
- `README.md` Results table: keep `/`, note fuzzy.
- No ADR. Decision small enough; CONTEXT.md update + `feat(filter)` commit covers it.

### CHANGELOG slice (v0.3.0)

```
### Changed
- Results filter (`/`) now uses fuzzy subsequence matching (`nucleo-matcher`) instead of plain substring. Each column is scored independently, row scores are combined, and matched graphemes render bold + underlined in the theme's focused-border color. Ranking uses the combined row score, with ties broken by original row order. Empty query restores unfiltered view. V6 still holds: filter operates only on the current page; PgDn re-applies the committed query to each fetched page.

### Added
- `nucleo-matcher` 0.3 dependency (MPL-2.0, pure Rust).
```

## 3. v0.3.1 — Hint bar

### Goal
One reserved row above the status bar shows context-aware keybindings: the active mode's top bindings on the left, a fixed palette suffix `<sp> cmd  ? help` on the right. Toggleable via config.

### Architecture

```text
┌─ Explorer ─┬─ Query ──────────────────────────┐
│            │                                  │
│            ├─ Results ────────────────────────┤
│            │                                  │
└────────────┴──────────────────────────────────┘
 / filter  v cell  yc copy  PgDn next            │  <sp> cmd  ? help   ← hint row (new)
 NORMAL · pg-prod · idle · 2 rows in 14ms                                ← status bar
```

### Components

```rust
// src/mode.rs — existing struct, extend ordering rule
pub struct KeyBinding {
    pub key: &'static str,
    pub label: &'static str,
}
// Convention enforced by docs + bindings()-audit test: most-important-first.
// Hint bar takes the first N that fit; help overlay shows all.

// src/hint_bar.rs (new)
pub struct HintBar;
impl HintBar {
    pub fn render(
        f: &mut Frame,
        rect: Rect,
        mode_hints: &[KeyBinding],     // from active ModeHandler::bindings()
        theme: &Theme,
    );
}

const PALETTE_SUFFIX: &str = "<sp> cmd  ? help";
```

Render algorithm:
1. Compose left = mode bindings as `"<key> <label>"` joined by two spaces; measure char width.
2. Right = `PALETTE_SUFFIX`.
3. If `left.width + 4 + right.width <= rect.width` → render both with gap-fill between, separator `│` in `theme.hint_bar.separator`.
4. Else if `left.width <= rect.width` → render left only.
5. Else → drop trailing mode bindings one at a time until fits; if even one binding too long, render `"…"`.
6. Width < 40 cols → emit empty row (still reserved if config on; or skip if `auto_hide_narrow = true`).

### Config

```toml
# ~/.sqrit/config.toml — new optional table
[hint_bar]
enabled = true            # default true on first run
auto_hide_narrow = false  # true → omit row entirely when width < 40
```

Missing `[hint_bar]` → defaults (enabled).

### Theme schema addition

```toml
# ~/.sqrit/themes/<name>.toml — new optional section
[hint_bar]
bg = "#1a1b26"
fg = "#9aa5ce"      # default body color
key = "#7aa2f7"     # accent for the key letter (e.g. "/" in "/ filter")
separator = "#414868"
```

Missing → per-field fallback:
- `bg` ← `panel.bg`
- `fg` ← `panel.fg`
- `key` ← `accent`
- `separator` ← `border.fg`

Five bundled themes updated to include the section in the same v0.3.1 commit; user-authored TOMLs without it still render correctly via fallback.

### Layout impact (`src/app.rs`)

```rust
let chunks = Layout::vertical([
    Constraint::Min(0),                                                  // main 3-pane
    Constraint::Length(if config.hint_bar.enabled { 1 } else { 0 }),     // hint row
    Constraint::Length(1),                                               // status bar
]).split(frame.size());
```

`enabled = false` → zero-height row → no overhead. Config read at startup; no runtime toggle in v0.3.1.

### Error handling
- Bindings list empty → render only palette suffix.
- Theme `[hint_bar]` section partial → per-field fallback.
- Terminal width < 20 → render empty row, no panic.

### Testing
- `tests/hint_bar.rs`:
  - Render at widths 120 / 80 / 60 / 40 → assert truncation order
  - `auto_hide_narrow = true` + width 30 → row absent
  - Config disabled → main 3-pane area gets the extra row back
- `tests/mode_bindings_audit.rs`: existing test extended to ensure every mode handler returns a non-empty bindings list (hint bar needs at least one entry to fall back from).

### ADR 7 (new `docs/adr/0007-hint-bar.md`)
- Decision: extra reserved row above status, mode bindings left + palette suffix right.
- Alternatives:
  - Replace status bar (A) — rejected: error message width squeezed unpredictably.
  - `<space>`-only popup (C) — rejected: diverges from "always-visible" goal.
- Consequences: every new mode handler must order `bindings()` most-important-first.

### Docs

| Doc | Change |
|---|---|
| `CONTEXT.md` | New "Hint Bar (v0.3)" entry; update "Status Bar" entry to note adjacent row above it. |
| `README.md` | New "Hint Bar" subsection with config snippet and screenshot. |
| `CLAUDE.md` | New invariant V10: "Hint bar bindings come from `ModeHandler::bindings()` only — never inline strings." |

### CHANGELOG slice (v0.3.1)

```
### Added
- Hint bar: a single-row, mode-aware keybinding hint above the status bar. Left lists the top bindings of the active mode (sourced from `ModeHandler::bindings()`); right always shows `<sp> cmd  ? help`. Truncates gracefully on narrow terminals.
- `[hint_bar]` config section in `~/.sqrit/config.toml` (`enabled`, `auto_hide_narrow`).
- `[hint_bar]` optional theme section (`bg`, `fg`, `key`, `separator`); missing fields fall back to existing palette colors.
- ADR 7: hint bar layout.
```

## 4. v0.3.2 — Schema browser

### Goal
Expose all schema-object kinds each backend supports — indexes, triggers, functions, procedures, sequences, materialized views — under a namespace-aware tree (PostgreSQL schemas, MySQL databases, SQLite single implicit namespace).

### Architecture
Two structural changes:

1. **`Database::schema_info()` returns `Vec<Namespace>`** instead of flat `tables` / `views` collections.
2. **`TreeItem` becomes a generic node**: `Namespace { name, expanded } → Group { kind, expanded, count } → Object { name, expanded } → Column { … }`. Column is valid only under `Table` / `View` / `MaterializedView`.

### Types (`src/db/types.rs`)

```rust
pub struct SchemaInfo {
    pub namespaces: Vec<Namespace>,
}

pub struct Namespace {
    pub name: String,                            // "public", "analytics"; "" for SQLite
    pub tables: Vec<TableObject>,
    pub views: Vec<ViewObject>,
    pub materialized_views: Vec<ViewObject>,     // PG only; empty for MySQL/SQLite
    pub indexes: Vec<IndexObject>,
    pub triggers: Vec<TriggerObject>,
    pub functions: Vec<RoutineObject>,           // PG functions, MySQL functions; empty for SQLite
    pub procedures: Vec<RoutineObject>,          // MySQL procedures; empty for PG (PG calls these functions) and SQLite
    pub sequences: Vec<SequenceObject>,          // PG only
}

pub struct TableObject     { pub name: String, pub columns: Vec<ColumnInfo> }
pub struct ViewObject      { pub name: String, pub columns: Vec<ColumnInfo> }
pub struct IndexObject     { pub name: String, pub table: String, pub unique: bool }
pub struct TriggerObject   { pub name: String, pub table: String, pub event: String }
pub struct RoutineObject   { pub name: String, pub return_type: Option<String> }
pub struct SequenceObject  { pub name: String, pub last_value: Option<i64> }
```

### Per-adapter introspection

| Object | SQLite | PostgreSQL | MySQL |
|---|---|---|---|
| Tables | `sqlite_master WHERE type='table'` | `pg_tables` | `information_schema.tables` |
| Views | `sqlite_master WHERE type='view'` | `pg_views` | `information_schema.views` |
| Materialized views | — | `pg_matviews` | — |
| Indexes | `sqlite_master WHERE type='index'` | `pg_indexes` | `information_schema.statistics` |
| Triggers | `sqlite_master WHERE type='trigger'` | `pg_trigger JOIN pg_class` | `information_schema.triggers` |
| Functions | — | `pg_proc WHERE prokind='f'` | `information_schema.routines WHERE routine_type='FUNCTION'` |
| Procedures | — | `pg_proc WHERE prokind='p'` | `information_schema.routines WHERE routine_type='PROCEDURE'` |
| Sequences | — | `pg_sequences` | — |

System schemas always filtered out:
- PG: `pg_catalog`, `information_schema`
- MySQL: `mysql`, `information_schema`, `performance_schema`, `sys`

A user-toggleable system-schema filter is deferred.

### Tree shape (`src/explorer.rs` rewrite)

```rust
pub enum TreeItem {
    Namespace { name: String, expanded: bool },
    Group     { ns: String, kind: ObjectKind, expanded: bool, count: usize },
    Object    { ns: String, kind: ObjectKind, name: String, expanded: bool },
    Column    { ns: String, kind: ObjectKind, parent: String, name: String, data_type: String },
}

pub enum ObjectKind {
    Table, View, MaterializedView,
    Index, Trigger, Function, Procedure, Sequence,
}
```

Render rules:
- Single namespace (`namespaces.len() == 1`, typical for SQLite and single-DB MySQL) → namespace layer auto-collapsed; groups render at root.
- Empty groups hidden (no `Sequences (0)` row in SQLite).
- Group header shows count: `▾ Tables (12)`.
- Column expansion limited to `Table`, `View`, `MaterializedView`.

### Keybindings (Explorer, additive)

| Key | Object | Action |
|---|---|---|
| `s` | Table, View, MaterializedView | `SELECT * FROM <ns>.<obj> LIMIT 100` (now qualified) |
| `s` | anything else | no-op; status bar message |
| Enter | any expandable | toggle expand |
| `j` / `k` | — | existing move |

Definition viewer (`d` key) deferred.

### Data flow
1. Connect → spawned `schema_info()` task (unchanged dispatch path).
2. Result → `App.schema: Vec<Namespace>` → `ExplorerState::set_schema()`.
3. `set_schema()` resets expansion: namespaces collapsed (except when single), groups collapsed.
4. UI: `j`/`k` walk visible items, Enter toggles expand.

### Migration
- `connections.toml`: unchanged.
- No on-disk schema cache exists today (re-loaded per connect), so no invalidation step needed.
- v0.3.2 binary connecting to any DB the v0.2 binary connects to works without user action.

### Error handling
- Each introspection query independent. One kind failing (e.g. permission-denied on `pg_matviews`) → log warning; that group renders as `▸ Materialized Views (?)`; selecting it is a no-op with a status-bar message.
- Object name containing quote chars → quoted with adapter-specific quoting helper when building `SELECT *`.

### Testing
- `tests/schema_pg.rs`, `tests/schema_mysql.rs`, `tests/schema_sqlite.rs`: seed one of each kind, assert `schema_info()` returns expected counts and namespaces.
- `tests/explorer_tree.rs`: flatten of `TreeItem` at various expansion states; namespace auto-collapse for single-ns case.
- Integration: `s` on a table in a non-default namespace produces `SELECT * FROM "analytics"."events" LIMIT 100` (PG quoting).

### ADR 8 (new `docs/adr/0008-namespace-aware-introspection.md`)
- Decision: `SchemaInfo` becomes namespace-rooted; per-kind buckets; backends report what they support.
- Alternatives:
  - Flat with `kind` tag — rejected: render explosion on 200+ object DBs.
  - Per-kind separate trait methods — rejected: N round-trips, harder to cancel and harder to keep introspection atomic.
- Consequences: every adapter pays a fixed introspection cost on connect. Lazy per-kind load deferred — a single ADR 8 amendment will cover it if perf complaints arrive.

### Docs

| Doc | Change |
|---|---|
| `CONTEXT.md` | New glossary entries **Namespace** and **Object Kind**; rewrite **Explorer** entry. |
| `README.md` | New screenshot of the namespace tree; extended Explorer key table. |
| `CLAUDE.md` | New invariant V11: "`Database::schema_info()` returns a namespace-rooted result; never flat." |

### CHANGELOG slice (v0.3.2)

```
### Added
- Schema browser: indexes, triggers, functions, procedures, sequences, materialized views, shown under collapsible per-kind groups beneath each namespace (PostgreSQL schemas / MySQL databases). SQLite shows a single implicit namespace that auto-collapses to root. Counts in group headers.
- ADR 8: namespace-aware introspection.

### Changed
- `Database::schema_info()` returns `Vec<Namespace>` instead of flat tables/views. All three adapters updated. System schemas (`pg_catalog`, `information_schema`, `mysql`, `performance_schema`, `sys`) filtered out.
- `s` in Explorer generates `SELECT * FROM <ns>.<obj> LIMIT 100` for tables, views, and materialized views; no-op with status message on other object kinds.
```

### Branch & release ordering

```
main ─┬─ feat/T10-fuzzy-filter ──── PR ──┐
      │                                    ├─ tag v0.3.0 ─→ release.yml
      └────────────────────────────────────┘
      ┌─ feat/T11-hint-bar ───────── PR ──┐
      │                                    ├─ tag v0.3.1 ─→ release.yml
      └────────────────────────────────────┘
      ┌─ feat/T12-schema-browser ── PR ──┐
      │                                    ├─ tag v0.3.2 ─→ release.yml
      └────────────────────────────────────┘
```

Each ticket follows the existing T# pattern (one `feat/T#-…` branch, review commits, fix commits — same shape as T1–T9). `[Unreleased]` block holds entries between tags; promoted to `## [0.3.x] - YYYY-MM-DD` on cut. `Cargo.toml` version bumped in the release-prep PR for each minor.

### Cross-cutting test strategy

| Layer | What | Minor |
|---|---|---|
| Adapter unit (SQLite) | new introspection queries return correct shape | v0.3.2 |
| Adapter integration (PG/MySQL) | `just it` covers new objects via seeded DDL | v0.3.2 |
| Mode dispatch audit | `bindings()` non-empty for every mode, ordering stable | v0.3.1 |
| Hint bar render | width truncation, theme fallback, disabled config | v0.3.1 |
| Fuzzy filter | scoring, highlighting, PgDn re-apply | v0.3.0 |
| Theme parser | missing `[hint_bar]` section, partial fields | v0.3.1 |
| Config loader | unknown top-level keys ignored | v0.3.1 (one-off) |

`cargo test` is the v0.3.0 gate; `just it` is the v0.3.2 gate (PG/MySQL must be running locally).

### Risks & rollback

| Risk | Mitigation | Rollback |
|---|---|---|
| `nucleo-matcher` API churn | Pin minor in Cargo.toml; one matcher entry point in `src/filter.rs` | Revert v0.3.0 commit; v0.2 substring path is one git history away |
| Hint bar eats vertical space on small terminals | `auto_hide_narrow` config + `enabled = false` opt-out | Set `enabled = false` in config; no code revert needed |
| Schema introspection slow on huge PG (10k+ functions) | System schemas filtered; log durations | Lazy per-kind load via ADR 8 amendment; emergency: env flag `SQRIT_DISABLE_INTROSPECTION_KINDS=functions,sequences` |
| ADR 8 changes `schema_info()` shape | None outside repo — binary crate, no public library API | n/a |

- **T11 — Hint bar** (v0.3.1): sub-issues T11a config + theme schema, T11b layout, T11c ModeHandler ordering audit, T11d docs/ADR 7.
